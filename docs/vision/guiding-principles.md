# Guiding Principles

**Project:** Datapex Connect

**Version:** 1.0

**Status:** Living Document

---

# Purpose

These principles guide every product, architectural and engineering decision made within Datapex Connect.

They are intentionally technology-independent.

When trade-offs arise, these principles take precedence over convenience or implementation preference.

---

# 1. People Before Technology

Technology exists to strengthen relationships.

Every feature should help professionals build trust, communicate clearly and continue meaningful conversations.

Technology is the enabler, not the product.

---

# 2. Product Before Implementation

The product is defined by its vision, domain model and specifications.

Implementation is simply one expression of that product.

Different technologies may implement the same product without changing its intent.

---

# 3. Architecture Before Code

Good architecture creates freedom.

Investing in architecture allows implementation to evolve without disrupting the product.

Short-term convenience should never compromise long-term maintainability.

---

# 4. Content Before Presentation

Business knowledge belongs in structured content.

User interfaces should render information rather than own it.

Separating content from presentation allows the platform to evolve without redesign.

---

# 5. Simplicity Scales

Every feature introduces complexity.

Only features that provide meaningful value should exist.

The simplest solution that satisfies the problem is usually the correct one.

---

# 6. Trust Through Restraint

Professional software should communicate confidence.

Avoid unnecessary decoration, excessive animation and visual distraction.

Whitespace, typography and clarity create trust.

---

# 7. Design for Change

Assume the product will evolve.

Architectures should support future capabilities rather than optimise only for today's requirements.

Extension is preferred over replacement.

---

# 8. AI Assists. Humans Decide.

Artificial Intelligence is an engineering partner.

Humans remain responsible for:

* Product vision
* Business decisions
* User experience
* Architecture
* Quality

AI accelerates implementation but does not own product direction.

---

# 9. Specifications Drive Development

Every meaningful feature should begin with a Product Specification.

Specifications communicate intent.

Implementation follows.

Code should never become the primary documentation.

---

# 10. Review the Product, Not Just the Code

Users experience products.

Not source code.

Every iteration should be reviewed from the perspective of:

* User experience
* Business value
* Maintainability
* Product quality

before implementation quality.

---

# 11. Continuous Learning

Every project should improve the engineering process.

Lessons learned should become part of the playbook.

The objective is not simply to deliver software, but to continuously refine how software is designed and delivered.

---

# Final Principle

Build products that outlive their implementation.

Technologies change.

Frameworks evolve.

Programming languages come and go.

Clear thinking, sound architecture and well-defined products remain valuable.
